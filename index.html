Featured case study

PhisDefense SOC and Email Security Lab

View full case study

Defensive email security and SOC monitoring project built around a real mail security environment. It validates SPF, DKIM, a progressive DMARC policy ending in p=reject, DNSSEC, MTA-STS, TLS-RPT, SMTP security controls and SOC-style evidence handling through a public dashboard.

Live simulation — spoofed mail is rejected at the DMARC gate (p=reject), authenticated mail reaches the inbox.

270 security tests
190 SOC events
p=reject final DMARC policy
s2026 DKIM rotation selector
Email Security Blue Team SOC Linux Postfix Dovecot OpenDKIM OpenDMARC OpenARC Python Dash
Project objective

Build a defensive email security lab where controls were not only configured, but also tested, documented and validated with technical evidence.

  • Validate how a real mail environment reacts to spoofing and SMTP abuse scenarios.
  • Demonstrate defensive controls through logs, SMTP responses and dashboard evidence.
  • Transform a technical deployment into a public case study suitable for portfolio review.
Technical environment

The project was deployed on a real Ubuntu VPS and used a full email security stack.

  • Postfix as the main SMTP service.
  • Dovecot for IMAPS TLS validation.
  • OpenDKIM for DKIM signing and validation.
  • OpenDMARC for DMARC policy evaluation and rejection.
  • OpenARC for ARC-related authentication chain support.
  • Syslog as the main source of technical evidence.
Validated controls

The project validated email authentication, DNS-based controls and transport security. DMARC was applied progressively until reaching the final enforcement policy p=reject. DKIM rotation was handled separately through the selector s2026.

  • SPF pass and SPF fail validation.
  • DKIM valid, broken and absent scenarios.
  • DKIM selector rotation to s2026.
  • Progressive DMARC policy ending in p=reject.
  • External spoofing rejection through DMARC.
  • DNSSEC, MTA-STS and TLS-RPT documentation.
  • STARTTLS SMTP and IMAPS TLS validation.
SOC evidence and dashboard

Server events and controlled tests were processed into a sanitized SOC dataset and visualized through an interactive Python/Dash dashboard.

  • 270 total security tests aggregated in the project.
  • 190 SOC events processed and categorized.
  • Events covering SMTP AUTH failures, recipient unknown, open relay testing and TLS checks.
  • Email security events covering SPF fail, DKIM issues and DMARC spoofing rejection.
  • Dashboard views for KPIs, tables, event categories, SMTP security and reporting status.
Public version and privacy

The public version was prepared for portfolio use while protecting sensitive operational data from the original environment.

  • No private DKIM keys.
  • No credentials or .env files.
  • No complete raw logs or full server backups.
  • No sensitive IP addresses or internal operational data.
  • Only sanitized datasets and public evidence are included.
Professional value

This project demonstrates practical Blue Team skills through a full defensive workflow: configuration, validation, evidence collection, automation, dashboarding and technical communication.

  • Relevant for SOC Analyst L1, Junior Cybersecurity Analyst and Blue Team junior roles.
  • Shows ability to document and explain technical controls clearly.
  • Connects infrastructure security with SOC-style analysis and reporting.

IAM case study

IAM Operations and CyberIAM Monitoring Lab

View full case study

An end-to-end IAM operations lab built as a small identity control center: a real Keycloak identity provider, a Flask application protected with OpenID Connect, Python automation for identity inventory and privileged account review, and a Streamlit CyberIAM dashboard that turns login, failed-login and admin events into reviewable, audit-friendly evidence.

Identity & access flow — the login request is redirected to Keycloak, the issued token passes the RBAC gate into the protected profile, and login, privileged and failed-login events stream into the IAM monitoring dashboard.

4 lab users and roles
1 protected Flask app (OIDC)
1 privileged account detected
10 normalized IAM events
Keycloak OpenID Connect Flask Python Docker Compose Streamlit pandas Altair Keycloak Admin REST API IAM RBAC CyberIAM
Project objective

Build a local hands-on lab that simulates the daily workflow of an IAM Operations team: identity lifecycle visibility, access control validation and evidence-based reporting.

  • Configure a real Keycloak identity provider with a dedicated realm, OIDC client, users and roles.
  • Protect a Flask application with the OpenID Connect Authorization Code Flow.
  • Automate identity inventory, privileged account review and IAM event monitoring with Python.
Technical environment

The lab runs locally with Docker Compose and uses a full IAM stack on top of Keycloak.

  • Keycloak realm airbus-iam-lab with users, roles and an OIDC client.
  • Flask application protected with OpenID Connect Authorization Code Flow.
  • Role-Based Access Control with filtered lab roles: user, admin, auditor, support.
  • Professional IAM identity profile UI on a protected /profile route.
  • Docker Compose orchestration for Keycloak and the Flask app.
IAM automation and monitoring

Python scripts use the Keycloak Admin REST API to automate operational IAM tasks and to extract authentication and administrative events.

  • User and role inventory export.
  • Privileged account detection.
  • Missing lab role control.
  • Daily IAM operations report.
  • Keycloak login and admin event monitoring.
  • Normalized IAM event pipeline with severity classification.
CyberIAM dashboard and evidence

A Streamlit IAM/CyberIAM operations dashboard consolidates the generated evidence into operational views for review and reporting.

  • Executive KPIs and identity inventory.
  • Privileged account review.
  • Failed login review and admin events review.
  • CSV and Markdown evidence outputs for auditability.
Security and privacy

The lab was built with a security-first approach to secrets and token handling.

  • No secrets are displayed in the UI.
  • Client secrets remain in the local .env file and are not committed.
  • Raw tokens are not displayed.
  • Technical Keycloak roles are filtered from the profile UI.
  • The dashboard reads evidence files and does not require live credentials.
Professional value

This lab demonstrates practical IAM operations skills directly related to IAM IT Operations, CyberIAM monitoring and service management workflows.

  • Identity lifecycle visibility and RBAC validation.
  • Privileged account review and authentication event monitoring.
  • Admin event review and evidence-based reporting.
